# 模組 05｜使用者、決策鏈與 RACI

🔴 必做

## 目的

搞清楚 4 件事：

1. **誰用** — 系統的日常使用者
2. **誰買** — 預算掌控者、簽核者
3. **誰擋** — 可能反對的人（IT、法務、財務、現場主管）
4. **誰背** — 出事誰負責

很多提案會死在「誰擋」這一塊。

## 角色分類

### 使用者光譜

| 角色 | 關心什麼 | 怎麼說服 |
|---|---|---|
| 老闆 / CEO | KPI、ROI、競爭力 | 一頁效益總結 |
| 部門主管 | 部門目標、稽核、流程順暢 | 流程改善前後對比 |
| 第一線員工 | 會不會更好用、會不會被取代 | 操作演示、訓練計畫 |
| IT | 維運、整合、資安 | 架構圖、SLA、權限 |
| 財務 | 成本、現金流、攤提 | 報價拆項、付款條件 |
| 法務 | 合約、責任、退出 | 合約附錄、退出機制 |
| 採購 | 比價、合規、供應商風險 | 競品分析、SBOM |

## RACI 矩陣

每個關鍵活動都要標：

| 代號 | 意思 |
|---|---|
| R - Responsible | 實際執行 |
| A - Accountable | 最終負責（每項只能一個 A） |
| C - Consulted | 諮詢（雙向溝通） |
| I - Informed | 通知（單向告知） |

### 範例

| 活動 | 老闆 | IT 主管 | 部門主管 | 使用者 | 我們 |
|---|---|---|---|---|---|
| 需求確認 | I | C | A | R | R |
| 系統選型 | A | C | C | I | R |
| 部署上線 | I | A | C | I | R |
| 驗收 | I | R | A | C | C |
| 維運 | I | A | I | R | C |

## 決策鏈圖

把人放在組織圖位置上，並標出：

- 誰是 **Economic Buyer**（掏錢的人）
- 誰是 **User Buyer**（會用的人）
- 誰是 **Technical Buyer**（評估技術的人）
- 誰是 **Coach**（內部支持我們的人）
- 誰是 **Blocker**（潛在反對的人）

## 常見地雷

- ❌ 只跟一個窗口往來，不知道後面決策鏈
- ❌ 沒識別出反對者，提案到簽核才被擋
- ❌ 把使用者當作可接受任何改變的人，沒考慮抗拒
- ❌ 沒問「上次失敗的系統是誰主導的」→ 那個人現在還在，他會用過去經驗判斷你

## 必問問題

- 這個案子最後誰簽？需要幾個人同意？
- IT 部門對這類系統的態度？
- 第一線員工現在用什麼？換新系統他們會抗拒嗎？
- 過去類似的系統導入過嗎？結果如何？誰主導的？
- 預算來自哪個 BU？年度還是季度？

## 輸出物

1. **角色清單**：每個角色姓名、職位、關心什麼、可能擋什麼
2. **RACI 矩陣**：5-10 個關鍵活動的責任分配
3. **決策鏈圖**：含 Coach、Blocker 標記
4. **溝通計畫雛形**：每個角色用什麼方式 / 頻率溝通

## 反方審查問題

- 這份提案最後誰簽？你直接跟他談過嗎？
- 誰是這案的 Blocker？你有對策嗎？
- 如果 IT 主管說「我們不要再買新系統」，你怎麼回？
- 第一線員工抗拒怎麼辦？訓練算進報價了嗎？
