# 模組 12｜風險、假設、限制、責任邊界與反方審查

🔴 必做

## 目的

主動把可能炸的地方寫出來。
寫不出風險的提案，等於把風險全部留給未來自己背。

## 五個子題

### 1. 風險清單（Risk Register）

| ID | 風險 | 機率 | 影響 | 風險值 | 處理方式 | Owner |
|---|---|---|---|---|---|---|
| R-01 | 客戶 ERP 沒 API | 高 | 高 | 9 | 提前 PoC、改用 CSV | 我方 PM |
| R-02 | 第一線抗拒 | 中 | 高 | 6 | 強化訓練、Champion 制 | 客方主管 |
| R-03 | 雲端供應商價格上漲 | 低 | 中 | 3 | 合約鎖價、年度檢視 | 我方財務 |

機率 / 影響用 1-3 評分，相乘得風險值。前三高的要寫對策。

### 風險類別

| 類別 | 範例 |
|---|---|
| 技術風險 | 整合失敗、效能不足、技術選型過時 |
| 資料風險 | 髒資料、量爆炸、跨境合規 |
| 組織風險 | 抗拒、主管異動、優先順序變更 |
| 商務風險 | 預算砍、付款延遲、合約變更 |
| 法規風險 | 法規變動、認證未過、稽核 |
| 第三方風險 | 廠商倒、API 異動、停服 |
| 環境風險 | 災害、停電、網路中斷 |
| 進度風險 | 客戶端決策慢、跨部門協調 |

### 2. 假設（Assumptions）

我們**預設成立**才能完成的事項。
寫出來，避免事後爭執。

範例：
- 假設客戶 ERP 提供 RESTful API（已確認）
- 假設客戶提供 1 名專案窗口至少 50% 工作時間
- 假設客戶現場網路頻寬 ≥ 100 Mbps
- 假設客戶資料保留期符合本案設計
- 假設客戶在 N 天內回覆我們的需求 / 設計確認

### 3. 限制（Constraints）

不能改變的條件：

- 預算上限
- 上線時程（例：年底前）
- 法規 deadline
- 既有系統不能換
- 既有合約鎖死的技術選型
- 人力可用性

### 4. 責任邊界（Responsibility Boundary）

> 跟模組 05 的 RACI 概念有 overlap 但**不同層**：
>
> | 維度 | 模組 05 RACI | 模組 12 責任邊界 |
> |---|---|---|
> | 對象 | **專案活動**（誰負責訪談、誰負責驗收）| **可交付物 / 工作項**（誰負責 ERP 改動、誰負責現場施工） |
> | 時點 | 整個案件期間 | 簽約前釐清、寫進合約附錄 |
> | 目的 | 內部協作順暢 | 出事時誰承擔法律責任 |
> | 文件 | RACI 矩陣 | 合約 SOW |
>
> 簡單記：**RACI 是合作分工、責任邊界是法律邊界**。
> 一張寫不下就拆兩張。


| 工作項 | 我方 | 客方 | 第三方 |
|---|---|---|---|
| 需求訪談 | R | C | - |
| 系統設計 | R | A | - |
| 系統開發 | R | I | - |
| 既有 ERP 改動 | C | R | A |
| 現場網路改善 | - | A | R |
| 機台拉線施工 | I | A | R |
| 使用者訓練 | R | A | - |
| 上線後維運 | R | A | - |
| 資料正確性 | C | R | - |
| 法規認證 | C | A | R |

⚠️ 「客戶以為包含、我們以為不包含」的範圍灰色地帶，**事前用這張表釐清**。

### 5. 反方審查（Red Team Review）

模擬三種人來打我們的提案：

#### 客戶法務

- DPA / BAA 簽不簽？
- 智財權歸屬怎麼定？
- 退出條款？資料怎麼帶走？
- 違約金？不可抗力？
- 跨境傳輸合法基礎？
- 個資事故通報？
- 第三方 sub-processor 名單？

#### 客戶 IT / 資安主管

- 你的資安認證在哪？ISO 27001？SOC 2？
- Pen test 報告？
- 你們員工背景檢查？離職權限回收 SOP？
- 我們的資料會不會用於 AI 訓練？
- 整合 IAM / SSO 怎麼做？
- SBOM 提供嗎？

#### 客戶老闆 / 採購

- 為什麼選你不選 A 公司？
- 三年總成本（TCO）？
- 萬一你公司倒了我怎麼辦？
- 過去做過幾個類似案例？能聯絡嗎？
- 你的客戶 logo 在哪？
- 售後 SLA 違約金多少？
