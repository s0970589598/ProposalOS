# 模組 04｜作業現況、流程與服務藍圖

🔴 必做

## 目的

讓客戶 + 我們對「現在怎麼做」與「以後怎麼做」**達成同一個認知**。
沒有 As-Is / To-Be，後面寫的功能都會錯位。

## 必畫 3 張圖

### 1. As-Is 流程圖（現況）

不要美化，要寫真實的現況：
- Excel 來 Excel 去
- 用 LINE 傳照片
- 紙本簽核
- 一個人請假流程就斷
- 報表是會計每月手算

### 2. To-Be 流程圖（目標）

導入系統後的樣子：
- 哪些步驟被自動化
- 哪些步驟還是人做、但變更快
- 哪些角色不見了 / 新增了

### 3. 服務藍圖（Service Blueprint）

把流程分四層：

```
┌──────────────────────────────────────┐
│ Customer Actions（客戶 / 使用者做什麼）  │
├──────────────────────────────────────┤
│ Frontstage（看得到的系統行為）         │
├──────────────────────────────────────┤
│ Backstage（看不到的系統行為 / 後台流程）│
├──────────────────────────────────────┤
│ Support Processes（支援流程：IT、客服） │
└──────────────────────────────────────┘
```

## 必填項目

| 項目 | As-Is | To-Be |
|---|---|---|
| 觸發事件 | | |
| 涉及角色 | | |
| 時間長度 | | |
| 工具 / 系統 | | |
| 痛點 / 摩擦 | | |
| 失敗模式 | | |

## 常見地雷

- ❌ As-Is 只問一個人 → 不同部門看到的現況不一樣
- ❌ To-Be 全自動 → 實際上有人類決策節點不能跳
- ❌ 沒畫例外流程 → 例外才是真痛點
- ❌ 只畫快樂路徑 → 失敗、退貨、客訴的處理沒寫

## 必問問題

- 一個案件從進來到結束，要經過哪些人？
- 哪些步驟最常出錯？怎麼補救？
- 哪些步驟最花時間？為什麼？
- 哪些步驟靠某個特定的人才能做？他請假怎麼辦？
- 例外流程（客訴、退貨、緊急訂單）怎麼跑？

## 輸出物

1. **As-Is 流程圖**（含時間、角色、工具）
2. **To-Be 流程圖**（標出自動化點）
3. **差異對比表**：每步驟前後時間 / 人力 / 錯誤率
4. **服務藍圖**：前台、後台、支援流程分層

## 反方審查問題

- 你的 As-Is 訪談過幾個人？跨部門嗎？
- To-Be 跟客戶確認過嗎？
- 例外流程怎麼處理？沒寫的話是不是預設「沒例外」？
- 過渡期（同時跑 As-Is 和 To-Be）怎麼辦？
