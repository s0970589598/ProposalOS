# 模組 03｜問題定義與經營目標

🔴 必做

## 目的

把客戶嘴巴說的「需求」翻譯成「問題」，再對應到「老闆的 KPI」。
沒有對應到 KPI 的需求，就算做完客戶也不會覺得有價值。

## 三層結構

```
經營目標（老闆關心的 KPI）
   ↓
問題定義（為什麼現在做不到目標）
   ↓
需求（要做什麼來解決問題）
```

## 必填項目

### 經營目標（Business Goals）

| 目標類型 | 範例 | 衡量指標 |
|---|---|---|
| 營收增長 | 新事業、漲價、留客 | 營收、ARR、ARPU |
| 成本降低 | 人力、能耗、原料、停機 | 單位成本、總成本 |
| 效率提升 | 稼動率、產出、交期 | 稼動率%、交期天數 |
| 品質改善 | 良率、退貨率、客訴 | 良率%、客訴件數 |
| 風險控制 | 法規、資安、停機 | 事件數、罰款金額 |
| 決策品質 | 數據可見度、回應速度 | 報表時差、決策週期 |

### 問題定義（Problems）

每個問題必須有：
- **誰**遇到這個問題（角色、部門）
- **多久**遇到一次（頻率）
- **影響多大**（量化）
- **目前怎麼處理**（變通方案）
- **為什麼**這樣處理不夠（痛點）

### 需求（Requirements）

不要直接寫客戶嘴上說的解法。先確認解法 vs 問題的對應。

## 業務功能盤點 — 產銷人發財（跨產業通用）

🔴 訪談前先用這個框架盤點，**確保不會只挖到一個 function 的痛、漏掉其他 function**。

| 字 | 業務功能 | 典型痛點 | 對應系統可能 | 對應 KPI |
|---|---|---|---|---|
| **產** | 生產 / 製造 / 交付 | 稼動率低、停機、品質、效率 | MES、SCADA、OEE | 稼動率、良率、產出、交期 |
| **銷** | 銷售 / 行銷 / 通路 / 客服 | 預測不準、線索流失、客訴慢 | CRM、行銷自動化、客服系統 | 業績、轉換率、NPS、留存 |
| **人** | 人力資源 / 組織 | 招募慢、訓練失準、離職率高 | HRIS、Learning Platform | 離職率、訓練 hr、生產力 |
| **發** | 研發 / 創新 / 產品開發 | 開發週期長、智財管理、知識流失 | PLM、研發協作、文件管理 | 上市時間、專利、開案成功率 |
| **財** | 財務 / 會計 / 資金 | 月結慢、預算失準、應收逾期 | ERP、財務系統、BI | 月結天數、現金週轉、毛利 |

### 怎麼用

**訪談前**：對照 5 個 function，問自己「客戶的問題集中在哪幾個？我訪談時是不是該 cover 不只一個？」

**訪談中**：用 5 字當 checklist，避免只挖到使用者熟悉的 function。例：跟廠長談容易談「產」，但他公司可能真正缺的是「銷」或「財」的數據可見度。

**寫提案時**：每個痛點標出它屬於哪個 function。同一系統可能跨多個 function（最好的提案通常是）。

### 跟人機料法環的差別

| 框架 | 問什麼 | 抽象層 | 適用 |
|---|---|---|---|
| **產銷人發財** | 客戶**哪個 business function** 在痛 | 業務層 | 所有產業 |
| **人機料法環** | 這個 function 的**作業環節**哪裡壞 | 作業層 | 偏製造、品管 |

兩個一起用：先用產銷人發財定位**痛在哪個 function**，再用人機料法環（或對應產業框架）拆**痛的細節**。

### 跨 function 的問題特別重要

| 跨 function 痛點 | 範例 |
|---|---|
| 產 × 銷 | 銷售預測 → 生產排程斷裂 |
| 產 × 財 | 生產成本算不準、單位成本失真 |
| 銷 × 財 | 應收帳款追蹤、業績獎金結算 |
| 人 × 產 | 排班、技能矩陣、工時管理 |
| 發 × 銷 | 新品上市、客戶 feedback 回研發 |
| 發 × 財 | 研發費用攤提、稅務 R&D 認列 |

**跨 function 痛點往往是價值最高的提案切入點**，因為單獨買 ERP / CRM / MES 都解不到。

## 把「假需求」翻譯成「真問題」

| 客戶嘴上說 | 翻譯後的真問題 |
|---|---|
| 「我要一個 AI 系統」 | 「我每月花 X 小時整理報表，能不能變 0」 |
| 「我要 dashboard」 | 「老闆每次問我同樣的問題，我都要查 1 小時」 |
| 「我要 LINE 通知」 | 「設備異常 3 小時後我才知道，已經損失 10 萬」 |
| 「我要做 ESG」 | 「客戶 / 銀行要求 ESG 報告，沒做拿不到單 / 貸款」 |

⚠️ **客戶嘴上的解法不一定對應真痛點**。要追問「為什麼」3-5 次。

## 必查項

- 老闆 / 部門主管的年度 KPI 是什麼？
- 這個系統做完，KPI 會怎麼動？
- 如果動不了，老闆會買單嗎？

## 輸出物

1. **問題 → KPI 對應表**：每個問題對應到至少一個經營目標
2. **量化痛點表**：每個問題的頻率 × 單次成本 = 年成本
3. **效益假設**：解了之後預期改善多少（要標假設，不要當承諾）

## 反方審查問題

- 客戶說的這個「需求」，背後的真問題是什麼？
- 如果不做，會發生什麼？
- 你的效益數字是怎麼算的？假設可信嗎？
- 老闆 / 採購 / 使用者三方對這個問題的看法一致嗎？
