# 模組 06｜資料來源、資料治理與系統整合

🔴 必做

## 目的

資料這層搞錯，後面什麼都對不齊。
這個模組要回答：**資料從哪來、誰擁有、怎麼治理、怎麼出去**。

## 四象限

```
                  資料新鮮度
                    高
                    |
   即時整合 ────────┼──────── 即時 stream
   （API / Webhook）|         （MQTT / Kafka）
                    |
   ─────────────────┼─────────────────  資料量
                    |
   批次整合          |         批次 stream
   （ETL / CSV）     |         （S3 / DB dump）
                    |
                    低
```

依資料量 × 新鮮度，整合方式不一樣。

## 必填項目

### 資料來源盤點

| 資料項 | 來源系統 | 擁有者 | 格式 | 更新頻率 | 量級 | 敏感度 |
|---|---|---|---|---|---|---|
| 訂單 | ERP | 業務部 | DB | 即時 | 萬筆/日 | 中 |
| 設備狀態 | PLC | 廠務 | Modbus | 1 秒 | 千次/秒 | 低 |
| 員工資料 | HR | 人資 | API | 日 | 千筆 | 高（個資） |

### 資料治理

| 維度 | 內容 |
|---|---|
| 擁有權 | 哪個部門 / 角色擁有這份資料 |
| 存取權 | 誰可以讀、誰可以寫、誰可以匯出 |
| 保留期 | 保留多久、何時刪除 |
| 主資料管理 | 同一個客戶 / 員工 / 產品在不同系統的 ID 對應 |
| 資料品質 | 缺失值、異常值、重複值的處理 |
| 稽核紀錄 | 誰在何時動了哪筆資料 |

### 整合方式

| 方式 | 適用情境 | 注意點 |
|---|---|---|
| 即時 API | 操作型即時資料 | 限流、重試、冪等 |
| Webhook | 事件驅動 | 失敗重送、簽章驗證 |
| 批次 ETL | 報表、分析 | 排程、失敗告警、增量 vs 全量 |
| 訊息佇列 | 異步處理、削峰 | 順序保證、毒訊息處理 |
| 檔案交換 | 老系統、跨組織 | 命名規則、加密、清檔 |
| DB 同步 | 大量、低延遲 | CDC、衝突處理 |

## 常見地雷

- ❌ 假設客戶 API 一定有 → 很多老系統只能匯出 CSV
- ❌ 沒算資料量 → 上線後爆掉
- ❌ 沒處理重複資料 → 同一個客戶在 3 個系統 ID 不一樣
- ❌ 沒寫資料保留期 → 個資法、GDPR 罰錢
- ❌ 沒寫主資料管理 → 報表全錯
- ❌ 假設客戶會給乾淨資料 → 90% 案件資料都是髒的

## 必問問題

- 這份資料是誰負責的？停線了找誰？
- 過去 12 個月有沒有資料事故？怎麼解的？
- 客戶 / 員工 / 產品 ID 在不同系統是同一個嗎？
- 資料保留多久？老資料怎麼處理？
- 有沒有現存的資料字典 / schema 文件？
- 個資、營業祕密、機敏資料的分類怎麼定？

## 輸出物

1. **資料來源清單**（含格式、頻率、量級、敏感度）
2. **資料流程圖**（從來源到目標的路徑）
3. **資料治理表**（擁有權、保留、稽核）
4. **整合方式表**（每個來源用哪種方式）
5. **資料字典初稿**

## 反方審查問題

- 你怎麼知道客戶的資料是乾淨的？驗過嗎？
- 量級你算過嗎？尖峰時段呢？
- 資料如果整合失敗，客戶端要做什麼？
- 個資怎麼脫敏？誰能看明文？
- 我們要拿到哪些資料的所有權？合約有寫嗎？
