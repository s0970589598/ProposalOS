# 模組 08｜系統架構、功能需求與非功能需求

🔴 必做

## 目的

把前面 7 個模組蒐集到的內容，**翻譯成系統可建構的規格**。
這個模組不是炫技，是讓決策者、IT、法務都能各看各的。

## 四層架構（TOGAF Domain）

```
┌─────────────────────────────────────────────────────┐
│ 業務架構（Business Architecture）                    │
│ - 業務能力、組織、流程                                │
├─────────────────────────────────────────────────────┤
│ 資料架構（Data Architecture）                        │
│ - 資料實體、流向、儲存、生命週期                      │
├─────────────────────────────────────────────────────┤
│ 應用架構（Application Architecture）                 │
│ - 應用元件、服務、整合                                │
├─────────────────────────────────────────────────────┤
│ 技術架構（Technology Architecture）                  │
│ - 基礎設施、網路、安全、部署                          │
└─────────────────────────────────────────────────────┘
```

每層都要有一張圖，不要混在一起畫。

## 功能需求（Functional Requirements）

### 用 User Story 寫，不要用「系統要 XX」

❌ 系統要有訂單管理功能
✅ As a 業務助理，我要能在訂單建立後 1 秒內收到通知，這樣我可以立刻分派給對應業務

### 格式

```
ID: FR-001
角色: 業務助理
場景: 訂單建立
動作: 收到通知並分派
驗收: 通知延遲 < 1 秒、可一鍵分派
優先級: P0 / P1 / P2
依賴: FR-002, FR-005
```

### 優先級

| 優先級 | 意思 |
|---|---|
| P0 | MVP 必有，沒做案子不能上線 |
| P1 | 第二階段，上線後 1-3 個月加 |
| P2 | 第三階段，未來考慮 |

不要全部寫 P0。

## 非功能需求（Non-Functional Requirements）

**這是最常被漏寫的部分**。

| 類別 | 項目 | 怎麼寫 |
|---|---|---|
| 效能 | 回應時間、吞吐量 | P95 < 500ms、1000 req/s |
| 可用性 | 系統可用率 | 99.5% / 99.9% / 99.95% |
| 可靠性 | RTO、RPO、MTBF | RTO 4h、RPO 1h |
| 擴展性 | 水平 / 垂直擴展 | 支援 10x 成長無需重構 |
| 安全性 | 加密、權限、稽核 | 對應模組 07 |
| 可維護性 | 變更頻率、部署時間 | 每週 1 版、零停機部署 |
| 可觀測性 | 日誌、指標、追蹤 | 全鏈路 trace、保留 90 天 |
| 相容性 | 瀏覽器、OS、整合 | Chrome / Safari / Edge 近 3 版 |
| 合規性 | 對應模組 07 | |
| 在地化 | 語系、時區、貨幣 | zh-TW、UTC+8、TWD |
| 可及性 | WCAG 2.1 AA | 視產業 |

⚠️ 寫數字要有依據，不要寫「99.99%」當 marketing。

## 系統架構圖必有元素

1. **External Actors**（外部使用者、第三方系統）
2. **System Boundary**（我們的範圍 vs 既有系統）
3. **Components**（各模組）
4. **Data Stores**（DB、快取、檔案、佇列）
5. **Integration Points**（API、webhook、ETL）
6. **Security Boundaries**（VPC、subnet、認證點）
7. **Deployment**（雲端、地端、混合）

## 常見地雷

- ❌ 只寫功能不寫 NFR → 上線後效能炸、客戶吵
- ❌ NFR 寫一堆但沒對應驗收 → 等於沒寫
- ❌ 全部 P0 → 報價會嚇死客戶
- ❌ 架構圖混在一張 → 看不懂哪層
- ❌ 沒有 System Boundary → 範圍蔓延
- ❌ 沒寫整合介面細節 → 開發階段才發現對方沒 API

## 必填項目

- 至少 4 張架構圖（業務、資料、應用、技術）
- 至少 10 條 User Story（含 P0/P1/P2 分布）
- 至少 8 條 NFR（每類至少一條）
- 整合介面表（與外部系統的對接清單）

- NFR 的數字怎麼來的？實測過嗎？
- 整合介面對方確認過嗎？
- 架構圖跟客戶 IT 對齊過嗎？
- P0 / P1 / P2 是誰排的？老闆認可嗎？
- 系統邊界外的東西，誰負責？合約有寫嗎？
