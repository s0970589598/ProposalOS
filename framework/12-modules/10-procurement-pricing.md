# 模組 10｜採購邏輯、商務策略與報價拆項

🔴 必做

## 目的

從「客戶怎麼買」往回設計報價，不是從「我們怎麼算成本」往前推。

## 採購邏輯先盤點

| 維度 | 內容 |
|---|---|
| 預算來源 | 哪個 BU、年度 / 季度 / 專案 |
| 預算金額 | 範圍、上下限 |
| 採購流程 | 詢價、比價、議價、簽核層級 |
| 採購週期 | 從詢價到簽約多久 |
| 付款條件 | 預付、分期、驗收後付、保留款 |
| 採購合規 | 三家比價、政府採購法、董事會、母公司核准 |
| 既有供應商管理 | 是否要加入供應商名冊、流程多久 |

⚠️ 不問清楚採購邏輯就報價，常常**價格對但流程不對**而吃單。

## 報價拆項原則

### ❌ 不要這樣

```
整套系統  500 萬
維護費    年 50 萬
```

客戶不知道 500 萬包含什麼，覺得很貴。

### ✅ 這樣拆

| 大項 | 細項 | 數量 | 單價 | 小計 |
|---|---|---|---|---|
| 一、軟體授權 | 後台系統年費 | 1 | 60 | 60 |
| | App 授權 | 50 user | 1.2 | 60 |
| 二、硬體 | Gateway | 10 | 5 | 50 |
| | 感測器 | 100 | 0.5 | 50 |
| 三、導入服務 | 訪談 + 需求 | 1 案 | 30 | 30 |
| | 系統設定 | 1 案 | 50 | 50 |
| | 整合開發 | 1 案 | 80 | 80 |
| | 教育訓練 | 3 場 | 5 | 15 |
| 四、第三方 | AI 服務年用量 | 1 | 20 | 20 |
| 五、運維 | 第 1 年支援 | 1 | 40 | 40 |
| | SLA Premium（選配）| 1 | 30 | 30 |
| **合計** | | | | **485** |

## 拆項好處

- 客戶可以**選配**（去掉某項減價）
- 客戶可以**分期**（硬體 vs 軟體 vs 服務分階段付）
- 客戶**比價更困難**（對方很難一對一比）
- 法務 / 採購**更容易過**（明細清楚）
- 將來**異動明確**（追加哪一項）

## 商務策略

### 定價依據

| 依據 | 用法 |
|---|---|
| 成本加價 | 算自己成本 × 利潤率 |
| 價值定價 | 看效益 ÷ ROI 推算 |
| 競品對標 | 看模組 02 的競品價格 |
| 客戶預算 | 模組 03 訪談到的數字 |

實務上是四個一起看，不是單選。

### 定價心理戰術（合法、合道德）

| 戰術 | 怎麼用 | 為什麼有效 |
|---|---|---|
| **Anchor（錨定）** | 先報最貴的「完整版」（如 520 萬）、再說「我推薦中間的標準版」（380 萬）| 客戶用 520 萬當參考點、380 萬感覺便宜 |
| **Decoy（誘餌）** | 三方案中刻意設計「基礎版」沒人會選、讓「標準版」看起來最划算 | 沒有對比、客戶會跟競品比；有對比、客戶在三方案內挑 |
| **Bundle（包裹）** | 硬體 + 軟體 + 服務 + SLA 全部一起報 | 客戶難一對一比競品、整體價值感更高 |
| **Tiered（階梯）** | Bronze / Silver / Gold 三層 SLA | 客戶自己選等級、不是砍我方總價 |
| **Loss aversion（損失規避）** | 強調「不做的年化損失」而非「做了的年化收益」| 損失感受比收益強 2-3 倍 |
| **Reference price（參考價）** | 先講競品 1500 萬、再講我方 380 萬 | 客戶覺得我方便宜 |
| **Charm pricing（畸零數）** | 報 398 萬不報 400 萬 | 心理上覺得是 300 多萬非 400 萬 — 大型 B2B 較少用、消費品常用 |
| **Foot-in-the-door（先小後大）** | 先賣 PoC 50 萬、上線後升級 380 萬正式版 | 已決策的客戶比新客戶好簽 |

### 紅線（不准用的）

- ❌ 假折扣（先漲再砍）
- ❌ 虛報「市場價」
- ❌ 偽造「限時」（明明不限時還說限時）
- ❌ 隱藏費用（後續才告知必加項）
- ❌ 利用客戶對技術不熟 over-promise

正派的心理戰術 = 幫客戶看清決策結構。
誤用的心理戰術 = 操弄客戶 = 短期成交、長期失信。

### 折扣策略

不要直接砍總價。要交換條件：

- 加長期合約 → 折
- 加大批量 → 折
- 接受我們的標準 SLA → 折
- 預付 → 折
- 允許我們用作案例 / 採訪 → 折
- 加附帶條件（資料使用權）→ 折

### 加值條款（提高總價而非降價）

- 加 SLA 等級
- 加優先支援
- 加客製化 hour
- 加額外訓練場次
- 加歷史資料移轉

## 用量上限與成本控管（SaaS / 雲端 / AI Token / API 計費）

🔴 **凡是按用量計費的項目都要寫**，不然成本會失控。

### 容易失控的計費類別

| 類別 | 典型情境 | 失控風險 |
|---|---|---|
| AI Token | LLM API（Claude / GPT / Gemini） | 一支 prompt bug 跑迴圈、客戶端濫用、prompt 長度爆炸 |
| API Call | 第三方 API（地圖、簡訊、金流、物流） | 重試風暴、無限 polling |
| 雲端運算 | Lambda invocations、ECS task 時數 | autoscaling 失控、無限執行 |
| 雲端儲存 | S3、DB IOPS、頻寬 | 日誌爆量、未清理舊資料 |
| 雲端流量 | 跨 region、CDN、egress | 設計錯導致大量 egress |
| 訊息服務 | SES、SNS、SQS、Push | 通知風暴、重送無限 |
| SaaS 訂閱 | per-seat、per-API call | 帳號清不掉、機器人帳號 |

### 必寫條款

| 項目 | 內容 |
|---|---|
| 用量基準 | 預估月用量（含尖峰）|
| 用量上限 | Hard limit（超過直接拒服）/ Soft limit（超過通知）|
| 超量計費單價 | 超出基準後的單價（建議比預付高 20-50%）|
| 客戶端責任 | 哪些用量算客戶端濫用（如 prompt 過長、polling） |
| 我方責任 | 哪些用量算我方 bug（如重試 bug、迴圈）|
| 監控告警 | 達 50% / 80% / 100% 自動通知 |
| 緊急中斷機制 | 超過 X 倍立刻熔斷 |
| 月度報表 | 用量明細給客戶 |

### AI Token 特有設計

```
應用層 prompt
   ↓
Rate Limit（每 user、每秒）
   ↓
Token Budget（每月上限）
   ↓
Cost Alert（達 80% 通知）
   ↓
Hard Cutoff（達 100% 拒服）
   ↓
人工解鎖 / 加購
```

### 三種計費模式給客戶選

| 模式 | 內容 | 適合 |
|---|---|---|
| 全包月費 | 含 X 量內，超量另計 | 客戶用量穩定 |
| 後付實支 | 按月實際用量帳 | 客戶用量波動大 |
| 預付儲值 | 預付 N 元抵 X 量 | 客戶想要明確上限 |

⚠️ 不要單一報「全包」— 客戶用量爆掉我方承擔不起。

### 常見地雷

- ❌ 報「無限使用」→ 客戶端開 100 個帳號跑機器人
- ❌ 沒設 Hard Limit → 一支 bug 燒掉一個月利潤
- ❌ 沒區分客戶責任 / 我方責任 → 出事爭執
- ❌ 沒給用量報表 → 客戶帳單炸了找不到原因
- ❌ AI Token 沒設 max_tokens → 一句 prompt 爆 100k token

## 付款條件

| 模式 | 內容 |
|---|---|
| 預付 | 簽約 100% |
| 三段付 | 簽約 30% / 階段交付 40% / 驗收 30% |
| 四段付 | 簽約 20% / 階段 30% / 上線 30% / 驗收 20% |
| 訂閱制 | 月 / 季 / 年付 |
| 保留款 | 驗收後留 10-20%，N 個月後付清 |

## 常見地雷

- ❌ 不問預算就報價 → 報太高或太低
- ❌ 報價含糊（「依實際工時計算」）→ 客戶不敢簽
- ❌ 沒寫「不包含」 → 後續免費追加
- ❌ 維護費比例失衡 → 業界通常是首年合約 15-20%
- ❌ 沒留漲價空間 → 雲端費用上漲變吃虧
- ❌ 沒寫匯率、稅金、報關 → 進口硬體後補不到錢

## 必寫條款

- 報價有效期（30 / 60 / 90 天）
- 範圍變更管控（CR 流程、單價）
- 匯率、稅金、運費歸屬
- 付款條件、發票條件
- 違約金、延遲利息
- 智財權歸屬
- 保密協議

## 反方審查問題

- 客戶預算多少？你怎麼知道的？
- 你的報價跟競品差多少？為什麼差？
- 採購流程多久？你估算的成交日對得上嗎？
- 範圍變更怎麼算？客戶簽前要不要看 CR 單價？
- 三年總擁有成本（TCO）你算過嗎？
