# 客戶端視角 — 從讀提案者角度反向設計

⚠️ ProposalOS 95% 是「寫提案者視角」、但提案**是寫給客戶讀**的。
這個檔補上「客戶端讀提案時在想什麼」。

## 客戶端的真實處境

提案不是被「閱讀」、是被**「快速掃描 + 重點挑出」**。

| 客戶角色 | 平均花多少時間 | 怎麼讀 |
|---|---|---|
| 老闆 / CEO | **2-5 分鐘** | 只看執行摘要、效益、總價 |
| 部門主管 | **15-30 分鐘** | 看自己部門相關章節、跳過架構 |
| IT / 資安 | **30-60 分鐘** | 跳到架構、SLA、SBOM、Pen Test |
| 法務 | **45-90 分鐘** | 跳到合約條款、DPA、退場、智財 |
| 採購 / 財務 | **20-40 分鐘** | 跳到報價、付款、TCO、競品對比 |

→ 沒有人**從頭讀到尾**。
→ 不同角色**讀不同章節**。
→ 章節**要能獨立讀懂**、不能假設前面已建立 context。

## 客戶 30 秒掃描注意力地圖

執行摘要那一頁、客戶眼睛真實掃描順序：

```
1 秒 ─ 標題（這份是給我們的？是哪家提案？）
   ↓
3 秒 ─ 「投資金額」（不論寫得多藝術、客戶眼睛會自己找錢）
   ↓
5 秒 ─ 「時程」（多久要付出 / 多久能拿到）
   ↓
8 秒 ─ 「效益」（具體數字、不是形容詞）
   ↓
15 秒 ─ 看一張圖（最好是前後差異圖）
   ↓
30 秒 ─ 跳到自己最關心的章節
```

→ 執行摘要**必須**在這 30 秒內被「掃」過去還留下印象。

## 客戶端的 5 個「決定要不要繼續讀」訊號

| 訊號 | 正面 | 負面 |
|---|---|---|
| **第一句** | 講到客戶痛點 | 講我方多厲害 |
| **第一張圖** | 有數字、能對應現實 | 制式架構圖、看不出差別 |
| **第一個數字** | 對應客戶可信的 baseline | 不知道從哪來的「業界 30%」|
| **頁數** | 對應提案規模（Short 5 頁、Standard 20-30 頁、Enterprise 50-100 頁）| 小案 50 頁、大案 5 頁 |
| **客戶引述** | 同產業 reference | 沒有 / 都不是同產業 |

## 客戶端讀提案的 5 個常見「卡點」

### 1. 「為什麼選你不選 A 公司？」

客戶**心裡一定有對手**。提案內如果沒處理競品差異、客戶會自己 fill in 認知。
→ 對應[模組 02 競品分析](modules/02-competitors-alternatives.md)前移。

### 2. 「萬一你公司倒了我怎麼辦？」

中小規模供應商提案永遠的疑問。
→ 對應[強制檢查項 G 退出機制](8-mandatory-checks/G-exit-plan.md)。

### 3. 「你做過幾個 → 給我 reference」

客戶想跳過你的話術、直接打給做過的人。
→ 對應[強制檢查項 H 證據包](8-mandatory-checks/H-evidence-pack.md)。

### 4. 「這個風險你怎麼處理？」

客戶會盤點所有風險、想看我方是否預想到。
→ 對應[模組 12 風險](modules/12-risk-assumptions-red-team.md)、寫**前 5 大風險 + 對策**。

### 5. 「我不續約怎麼辦？」

法務的標準題。
→ 對應[強制檢查項 G 退出機制](8-mandatory-checks/G-exit-plan.md)。

## 客戶端的「踩雷紀錄」（從讀過爛提案的客戶聽到）

| 客戶說過的雷 | 我方對策 |
|---|---|
| 「投影片好多動畫、看不到重點」 | 簡報跟提案文件分離、文件保持資訊密度 |
| 「全部都是『我們很厲害』」 | 強調客戶痛點、不是強調我方 |
| 「數字寫得太誇張、像在騙」 | 用「假設」字眼、不要寫成承諾 |
| 「同樣的話換 5 種寫法」 | Pyramid Principle、每段一個結論 |
| 「不知道你們公司財務健全嗎」 | Enterprise 版含財務證明附錄 |
| 「我看不到你們有什麼別人沒有」 | Win Themes + Discriminators 明確 |
| 「報價只有總價、不知道在算什麼」 | 報價拆項 |
| 「假設條件全部模糊」 | 假設清單明確 |
| 「我們有問問題、得到答案是『以實際為準』」 | 給範圍、給選項、別含糊 |
| 「客戶 case 都是別國 / 別產業」 | 同產業 reference 優先 |

## 客戶端的 3 種典型決策模式

### 模式 1：理性派（評選委員、政府、金融）

- 完全按評分表
- Compliance Matrix 重要
- 強制條款不能漏
- 證據完整度比論述深度重要

→ 對應 [Shipley](methodologies/shipley-apmp.md) + [Compliance Matrix](../templates/compliance-matrix-template.md)。

### 模式 2：直覺派（中小企業老闆、創辦人）

- 5 分鐘讀完就決定
- 看效益、看人、看 case
- 不一定看技術細節
- 重視「能不能信任」

→ 對應 [Pyramid Principle / SCQA](methodologies/pyramid-principle.md) 開門見山 + 強 reference。

### 模式 3：謹慎派（資安、法務、IT）

- 找雷不找好處
- 任何一個漏洞就否決
- 過去被坑過、會問細節
- 看認證、看合約

→ 對應 [強制檢查項 D-G](8-mandatory-checks/) 全套完備。

→ **大型案件三種人都會出現**、要同時打到。

## 客戶端讀完提案後做什麼

| 動作 | 機率 | 我方準備 |
|---|---|---|
| 內部開會討論 | 高 | 提案要能「轉發」、不要假設我方在場解釋 |
| 找 reference 客戶打電話 | 中 | 事前準備好 3-5 個 reference 願意接電話 |
| 找競品比較 | 高 | Win Themes / Discriminators 明確 |
| 找法務 review | 高（大案）| DPA / SLA / 退場條款完整 |
| 砍價 | 高 | 報價有空間、不要一槍到底 |
| 提變更需求 | 中 | CR 流程明確 |

## 客戶會做什麼「不公開」的動作

| 隱藏動作 | 我方對應 |
|---|---|
| 把我方提案給對手看（問價）| 提案有 watermark、寫保密期限 |
| 反向工程我方報價 | 拆項合理、不要破綻 |
| 內部詢問「為什麼選你」 | Win Themes 要能被客戶內部 champion 轉述 |
| 評估「砍我方員工 / 找下游」風險 | 強調 IP / 客製化歸屬 |
| 找前員工 / 同行打聽我方 | 公司聲譽、員工流動率管理 |

## 反向工程：客戶贏的提案長什麼樣

從跟客戶詳聊發現「他們選哪家、為什麼」、抽出共同 pattern：

| 贏家共同點 | 對應 ProposalOS |
|---|---|
| 提案前已建立關係 | Capture Plan |
| 訪談多次、深入痛點 | 模組 03 + JTBD |
| 給 2-3 方案讓客戶選 | 模組 09 |
| 同產業 ≥ 3 case | 證據包 |
| 退場條款主動寫清楚 | 強制檢查項 G |
| 報價拆項、能砍 / 能加 | 模組 10 |
| 提案說明會有來 | 流程階段 10 |
| 法務問題 48h 內回 | 溝通計畫 |

→ **這 8 個都做到 = 勝率 80%+**（依不同產業）。

## 我方的「客戶端訪談計畫」（建議內部做）

為了補這份文件的真實性、應該定期做：

| 動作 | 頻率 |
|---|---|
| 訪談**贏的客戶**「為什麼選我們」 | 每個案上線後 3 個月 |
| 訪談**輸的客戶**「為什麼選別人」 | 每個失敗案後 1 個月 |
| 訪談**評委 / 採購 / IT 主管** | 每年 5-10 個 |
| 把訪談發現寫進 ProposalOS | 每半年 review |

→ 這份檔現在是**從零組裝的客戶視角**、用真實訪談補強後才會準。
