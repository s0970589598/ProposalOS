# G｜退出機制

🔴 簽前必交

## 為什麼必有

客戶法務必問：「如果我們未來換廠商，資料怎麼帶走？」

沒有退出機制 = 客戶覺得自己被綁架 = 不簽。

寫好退出機制 = 反而讓客戶敢簽（因為知道有退路）。

## 三種退出情境

| 情境 | 觸發 |
|---|---|
| 客戶主動換廠商 | 合約期滿、不滿意 |
| 我方無法繼續服務 | 廠商倒、被收購、停產 |
| 雙方協議終止 | 重大爭議、不可抗力 |

每種情境都要設計退出流程。

## 退出機制必含元素

### 1. 資料匯出

| 項目 | 內容 |
|---|---|
| 範圍 | 所有客戶資料 + metadata + 設定 |
| 格式 | 開放格式（CSV、JSON、Parquet），不准專屬格式 |
| 完整性 | Schema、關聯、附件（檔案、圖片）一起 |
| 加密 | 傳輸與儲存加密 |
| 時程 | 從客戶提出到交付的天數 |
| 驗證 | 客戶確認資料完整無誤 |
| 費用 | 一次性匯出費（建議含在合約）|

⚠️ 提案要附**範例 schema** 讓客戶法務確認可被遷移。

### 2. 服務終止

| 項目 | 內容 |
|---|---|
| 通知期 | 至少 60-90 天 |
| 過渡期 | 30-90 天並行運作 |
| 唯讀模式 | 終止前可進入唯讀 |
| 帳號回收 | 我方人員權限歸零 |
| 設定備份 | 客戶端設定可帶走 |

### 3. 移轉協助

| 項目 | 內容 |
|---|---|
| 移轉文件 | Schema、API、整合說明 |
| 技術協助 | N 小時免費協助 / 計費協助 |
| 新廠商對接 | 配合新廠商 PoC、測試 |
| 資料說明會議 | 釐清欄位、邏輯 |

### 4. 我方資料清除

| 項目 | 內容 |
|---|---|
| 客戶資料完整刪除 | 含備份、log、暫存 |
| 刪除證明 | 出具 Certificate of Destruction |
| 時程 | 30-90 天內完成 |
| 例外保留 | 法規要求保留的項目（稅務、稽核） |
| 子處理者 | 通知並要求所有 sub-processor 刪除 |

### 5. 智財權與授權

| 項目 | 內容 |
|---|---|
| 客戶資料所有權 | 始終屬於客戶 |
| 客製化開發 | 看合約：歸客戶 / 歸我方 / 共有 |
| 平台核心 | 屬我方，授權客戶使用至終止 |
| 整合介接 | 文件交付給客戶 |
| 設定參數 | 視為客戶資料 |

## 提案中要附

| 章節 | 內容 |
|---|---|
| 退出機制總覽 | 三情境、流程圖 |
| 資料匯出規格 | Schema、格式、範例 |
| 服務終止程序 | 時程、責任、產出物 |
| 移轉協助方案 | 含費 / 不含費項目 |
| 資料刪除政策 | 法規例外、證明 |
| 智財權條款 | 各類資產歸屬 |
| 退出成本估算 | 讓客戶心裡有底 |

## 為什麼這對成交有幫助

很多人以為寫退出機制 = 提示客戶離開。
**實際上是反過來**：

- 客戶法務一看：「他們敢寫退出，表示有信心、有規範」
- 客戶採購：「萬一你倒了我也救得回來，我才敢買」
- 客戶老闆：「不是綁架式合約，比較放心」

## 常見地雷

- ❌ 沒寫退出機制 → 客戶法務直接退件
- ❌ 「資料可匯出 CSV」沒講 schema → 客戶不知道能不能用
- ❌ 沒講通知期 → 客戶覺得被綁
- ❌ 移轉協助全收費 → 留壞印象
- ❌ 沒寫子處理者刪除 → GDPR 卡死
- ❌ 客製化開發歸屬不清 → 換廠商時打官司

## 反方審查問題

- 我們不續約，資料多久能拿到？
- 你的匯出格式我們新廠商讀得了嗎？
- 你刪除資料怎麼證明？
- 移轉協助的人天怎麼算？
- 我們客製化的程式碼，能帶走嗎？
- 你公司倒了，誰負責退場？
