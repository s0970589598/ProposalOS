# F｜RTO / RPO、備份還原、版本回滾

🔴 簽前必交

## 為什麼必有

寫「我們有備份」是嘴砲。
客戶要的是：

- 壞了多久能回來？（RTO）
- 最多會掉多少資料？（RPO）
- 換新版炸了能退回去嗎？（回滾）

這三個數字寫不出來 → 客戶 IT 不放心。

## RTO / RPO 定義

```
           事故發生
              │
   ◄─────────┤─────────►
       RPO   │   RTO
   (最後備份) │ (恢復完成)
              │
   可接受的   │   可接受的
   資料遺失   │   服務中斷
```

- **RPO（Recovery Point Objective）**：最大可接受的資料遺失時間
- **RTO（Recovery Time Objective）**：最大可接受的服務中斷時間

### 範例值

| 系統類型 | RTO | RPO |
|---|---|---|
| 即時交易（金融、電商核心） | 5 min | 0 sec |
| 客戶營運系統 | 1-4 hr | 15 min |
| 內部營運系統 | 4-24 hr | 1 hr |
| 報表、BI | 1-3 day | 24 hr |
| 歷史歸檔 | 1 週 | 1 天 |

RTO / RPO 越短，成本越高。要跟客戶談「值不值」。

## 備份策略

### 3-2-1 原則

- **3** 份資料拷貝
- **2** 種不同媒體
- **1** 份異地

### 必寫項目

| 項目 | 內容 |
|---|---|
| 備份範圍 | 哪些資料、哪些系統 |
| 備份頻率 | 全量 / 增量 / CDC |
| 備份保留 | 多久、輪替策略 |
| 備份位置 | 同 region / 跨 region / 跨雲 |
| 備份加密 | At-rest、In-transit |
| 備份驗證 | 自動驗證可還原 |
| 還原演練 | 多久演練一次、紀錄 |

⚠️ 「有備份」≠ 「能還原」。**必須定期演練還原**。

## 版本回滾（Rollback）

| 層級 | 回滾方式 |
|---|---|
| 應用程式碼 | Blue-Green、Canary、版本標籤 |
| 資料庫 schema | Migration 必有 down() |
| 資料 | Point-in-time recovery、快照 |
| 基礎設施 | IaC 版本標籤、Terraform state |
| 設定 | Config 版本控管 |

### 部署策略

| 策略 | 回滾速度 | 成本 |
|---|---|---|
| Blue-Green | 秒 | 高（兩套並存） |
| Canary | 分鐘 | 中 |
| Rolling | 分鐘 | 低 |
| Recreate（直接換）| 慢 / 不可回滾 | 低 |

### 不可回滾的情境

- DB schema 已有 destructive change（刪欄位、刪表）
- 已對外發出不可撤回事件（通知、付款、訂單）
- 已對接的第三方系統已收到資料

這些情境必須**前向修復**而非回滾，事前要識別。

## 災難復原（DR）

| DR 等級 | 內容 | 成本 |
|---|---|---|
| Pilot Light | 最小化備援、災時擴展 | 低 |
| Warm Standby | 縮小版備援、災時切換 | 中 |
| Multi-Site Active-Active | 全規模並行 | 高 |

## 提案中要附

| 項目 | 內容 |
|---|---|
| RTO / RPO 承諾 | 寫進 SLA |
| 備份方案 | 範圍、頻率、保留、位置 |
| 還原演練計畫 | 頻率、文件、紀錄 |
| 部署策略 | 哪種、回滾時間 |
| Migration 回滾計畫 | DB 變更如何回滾 |
| DR 等級 | Pilot Light / Warm / Active |
| 不可回滾情境清單 | 事前識別 |

## 常見地雷

- ❌ 寫 RTO 5 min 但沒做 Active-Active → 做不到
- ❌ 備份沒驗證 → 還原才發現備份壞
- ❌ 跨 region 備份沒測過 → 真出事切不過去
- ❌ DB migration 沒 down() → 改錯回不去
- ❌ Canary 沒設自動 rollback 條件 → 沒人盯就炸

## 與其他模組的關係

- ← 模組 11：驗收、SLA、維運
- ← 強制檢查項 D：Secure SDLC 的 Operations 階段
