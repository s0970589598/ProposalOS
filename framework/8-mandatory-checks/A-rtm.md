# A｜RTM 需求追蹤矩陣（Requirements Traceability Matrix）

🔴 簽前必交

## 為什麼必有

把「痛點 → 需求 → 功能 → 驗收 → 報價 → 交付項目」串成一條鏈。
沒有 RTM，提案中段就會出現「我以為有 X」的爭執。

## RTM 結構

| 痛點 ID | 痛點描述 | 業務目標 | 需求 ID | 需求描述 | 功能 ID | 功能描述 | NFR | 驗收 ID | 驗收標準 | 報價項 | 階段 | 狀態 |
|---|---|---|---|---|---|---|---|---|---|---|---|---|
| P-01 | 異常 3h 後才知 | 降低停機損失 | R-01 | 即時告警 | FR-05 | LINE 通知 | NFR-02 | AC-08 | < 5min 送達 | 報價 II-2 | P0 | 已驗收 |

## 反向追溯

RTM 必須**雙向可追**：

- **痛點 → 功能**：每個痛點都有對應功能
- **功能 → 痛點**：每個功能都對應一個痛點

❌ 沒對應痛點的功能 → 多做的、要砍
❌ 沒對應功能的痛點 → 沒解到、要補

## 維護

RTM 不是寫完就鎖。
從 **需求訪談** 到 **正式上線** 全程更新。

| 階段 | 動作 |
|---|---|
| 需求訪談 | 建立「痛點 → 需求」 |
| 設計 | 加入「功能 → NFR」 |
| 報價 | 對應「報價項」 |
| 開發 | 更新「狀態 = 開發中」 |
| 測試 | 更新「狀態 = 測試中」 |
| 驗收 | 更新「狀態 = 已驗收」 |
| 上線 | 更新「狀態 = 已上線」 |

## 反方審查問題

- 客戶說的痛點，每個都有對應功能嗎？
- 你列的每個功能，都能指回某個痛點嗎？
- 報價的每一項，都能對應到 RTM 嗎？
- 客戶要求變更時，RTM 多久能更新？
