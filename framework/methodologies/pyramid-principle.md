# Pyramid Principle / SCQA

Barbara Minto（McKinsey 前顧問）的結構化思考與寫作方法。
**McKinsey、BCG、Bain 等顧問業內部寫作標準**。

我們的執行摘要、推薦方案說明、回信客戶都該用這套。

## 核心 1：金字塔結構（Pyramid）

```
        結論（1 個）
            │
   ┌────────┼────────┐
   理由 A   理由 B   理由 C  （3-5 個支持）
     │       │       │
  └─┬─┘   └─┬─┘   └─┬─┘
 證據 證據 證據 證據 證據 證據  （資料 / 案例 / 數字）
```

### 三條規則

1. **每個層次的結論支援上一層、是下一層的摘要**（垂直邏輯）
2. **同一層的結論互斥（MECE）**（水平邏輯）
3. **同一層的結論順序有邏輯**（時間、結構、重要性）

### MECE 是什麼

**M**utually **E**xclusive, **C**ollectively **E**xhaustive — 互斥但窮盡。

| 不 MECE | MECE |
|---|---|
| 客戶分：大企業、製造業、新創 | 客戶分：員工 < 50 / 50-500 / 500+ |
| 痛點：人力、成本、稼動率 | 痛點：產 / 銷 / 人 / 發 / 財 |

不 MECE 的分類 → 同一件事被算兩次 / 漏算。

## 核心 2：SCQA 開場架構

**寫文章 / 簡報 / Email 的開頭必用**：

| 字 | 意思 | 範例 |
|---|---|---|
| **S**ituation（情境）| 雙方都同意的事實 | 「貴公司在 2027 年要上市櫃」 |
| **C**omplication（衝突）| 一個衝突 / 變化 | 「上市櫃必須有 6 個月可信食安數據」 |
| **Q**uestion（問題）| 衍生的問題 | 「在 2026 年底前如何上線可信食安系統？」 |
| **A**nswer（答案）| 你的建議 | 「我們提出 6 個月內可完成的 IoT 監控方案」 |

→ Answer 後面接金字塔結構展開細節。

### SCQA 變形

| 形態 | 用在 |
|---|---|
| SCQ | 開場引入問題、答案後面才講 |
| QSCA | 直接拋問題、再講背景 |
| SAQC | 開場給答案、後面解釋 |

GoodLinker 提案推薦 **SCQA**：先建立共識 → 點出衝突 → 引出問題 → 給答案。

## 核心 3：水平 / 垂直邏輯

### 垂直邏輯（每層是上層的摘要）

```
結論：標準版是最適合本案的方案
   ↓ 為什麼？
   ├── 預算契合（在王副總可簽範圍）
   ├── 時程契合（6 個月對應上市櫃 deadline）
   └── 功能契合（解到 P0 8 條痛點全部）
        ↓ 為什麼預算契合？
        ├── 380 萬 < 500 萬上限
        ├── 包含 1 年支援、無隱藏費用
        └── ROI 5-7 個月、董事會易過
```

### 水平邏輯（同層 MECE）

「預算 / 時程 / 功能」是 MECE 嗎？
- 互斥：✓（不會重疊）
- 窮盡：可能還有「風險」、「相容性」等

→ 寫的時候每層自問：**這 3 點互斥嗎？窮盡嗎？**

## 套用 ProposalOS 各章節

### 執行摘要（一頁）

用 SCQA + 金字塔：

```
[S] 你們 2027 要上市櫃、客戶要求 2026 起有溫度履歷
[C] 但凌晨無人值守時的斷鏈占 80% 損失
[Q] 如何在 2026 底前上線、解凌晨問題、且預算可控？
[A] 我們建議標準版 380 萬 IoT 監控

理由 1：時程契合（6 個月）
理由 2：解到 P0 8 條痛點
理由 3：預算可控（單一董事會可簽）

每個理由展開支持證據...
```

### 推薦方案說明段

```
結論：推薦標準版

支持 1：契合上市櫃時程
   證據 1.1：6 個月可上線（vs 國際大廠 3 年）
   證據 1.2：6 個月 + 上線後 6 個月稽核資料 = 2027 Q3 送件可用

支持 2：契合預算結構
   證據 2.1：380 萬 < 王副總可簽 500 萬上限
   證據 2.2：付款拆 3 段（30/40/30）、現金壓力小

支持 3：契合 P0 痛點解法
   證據 3.1：FR-01~09 解到 8/8 P0 痛點
   證據 3.2：完整版有 AI、本案非必要
```

### 反方審查回答

用金字塔回 — **結論先講、再講為什麼**：

```
❌ 不好的回答
「嗯這個我們會考慮、也許可以這樣處理、我們有幾個 case 大概有做過、可能可以...」

✅ 用金字塔
「會。
 理由 1：我們合約含上市櫃稽核配合 N 小時。
 理由 2：技術上系統設計符合 ISO 22000、稽核資料可即時匯出。
 理由 3：過去做過 X、Y 兩個食品上市櫃案、reference 可聯絡。」
```

## 反 Pyramid Principle 的常見錯誤

| 錯誤 | 為什麼錯 |
|---|---|
| 從背景講到結論（流水帳） | 讀者 30 秒沒看到結論就走 |
| 同層 4-7 個 bullet | 認知負擔超過、記不住 |
| 同層不 MECE（有重疊）| 顯得邏輯不清 |
| 證據跟結論對不齊 | 讀者覺得作者沒想清楚 |
| 結論後接時間順序流水帳 | 重要性沒排好 |

## 一句話自檢

寫完一段、自問三個問題：

1. **What's the point?**（這段的結論是什麼？）
2. **Why?**（支持結論的 3-5 個理由是什麼？）
3. **So what?**（讀者讀完知道要做什麼？）

回答不出來 → 重寫。

## 跟 SCQA 對應的其他變體

| 框架 | 用在 |
|---|---|
| SCQA | 顧問業、提案 |
| Hero's Journey | 故事 / 行銷 |
| BLUF（Bottom Line Up Front）| 軍方、軍工 |
| PREP（Point-Reason-Example-Point）| 演講 |
| STAR（Situation-Task-Action-Result）| 面試 / case interview |

提案場景 → 用 **SCQA**。
口頭簡報 → **BLUF**。
故事 / 客戶情境 → **STAR**。
