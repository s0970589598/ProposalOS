# Jobs to be Done（JTBD）

Clayton Christensen 提出。
**訪談時挖到「真正想完成的工作」、而不是表面的「想要什麼功能」**。

## 核心句型

> **「客戶不買 1/4 鑽頭，他要的是牆上的洞、最終是想掛畫。」**

把客戶的需求拆 3 層：

```
Functional Job（功能性工作）
  我想要：「冷鏈不要斷鏈」

Emotional Job（情感性工作）
  我感受：「不想再被半夜叫起來、不想再被老闆罵」

Social Job（社會性工作）
  我形象：「想被董事會看到我有掌握廠務、想被客戶覺得我們專業」
```

→ 三層都打到的提案 = 最會贏。
→ 只打 Functional 的提案 = 客戶覺得「有道理但不急著做」。

## 訪談用的 JTBD 句型

```
When [情境]
I want to [Functional Job]
So I can [Emotional / Social Job]
```

範例：

> 「When 凌晨壓縮機跳掉
> I want to 5 分鐘內收到告警並能遠端確認狀況
> So I can 不被客戶罵、保住下個月的續約」

→ 比「我要溫度告警系統」深 10 倍。

## 4 個 JTBD 力場（Forces of Progress）

決定客戶會不會買的 4 個力：

```
   推 push                  拉 pull
   現況不滿  ────►    新方案吸引  ◄────
                   │
                   │
   現況慣性  ◄────  新方案焦慮  ────►
   inertia                  anxiety
```

| 力 | 範例 |
|---|---|
| **Push（推）** | 「斷鏈一次賠 80 萬、撐不下去了」 |
| **Pull（拉）** | 「GoodLinker 5 分鐘告警，省心」 |
| **Inertia（慣性）** | 「我們習慣用 Excel、改變很麻煩」 |
| **Anxiety（焦慮）** | 「新系統會不會也壞？訓練多久？」 |

成交 = Push + Pull > Inertia + Anxiety

提案策略 = **強化 Push/Pull、消除 Anxiety**（Inertia 較難改）。

## 跟 ProposalOS 整合

### 模組 03 問題定義 加 JTBD 層次

| 痛點 | Functional Job | Emotional Job | Social Job |
|---|---|---|---|
| P-01 斷鏈 | 5min 告警 | 半夜不再被叫醒 | 廠長覺得有掌控 |
| P-02 配送爭議 | 即時溫度紀錄 | 不再被客戶質疑 | 業務有面子 |
| P-04 客訴回追 | 30min 鎖定 | 客訴不再恐慌 | 品管專業形象 |
| P-05 上市櫃 | 6 個月資料 | 不再擔心 IPO 延後 | 董事會有交代 |

→ 提案文案要打**三層都**，不只 functional。

### 訪談時用 JTBD 問

❌ 「你要什麼功能」（會得到清單）
✅ 「上次 X 發生時，你最希望系統怎麼幫你」（會得到真實情境）

### 4 力分析應用於反方審查

每個提案做完問：

- Push 強嗎？客戶**為什麼現在做**的痛感夠強？
- Pull 強嗎？我們的方案**有讓人興奮的點**嗎（不只是功能完整）？
- Inertia：客戶**對既有系統的依戀**有處理嗎？
- Anxiety：客戶**對新系統的擔心**有化解嗎？

任一不夠 → 提案會被卡住。

## JTBD vs 傳統需求訪談

| 傳統 | JTBD |
|---|---|
| 「你要什麼？」 | 「你想完成什麼工作？」 |
| 焦點：功能清單 | 焦點：情境 + 動機 |
| 找 feature gap | 找 progress gap |
| 客戶聽得懂 | 訪談者要練 |
| 易產生「假需求」 | 接近真痛點 |

## 範例：明陽食品 JTBD 重寫

**傳統**：「明陽要冷鏈監控系統」

**JTBD 1**：
> When 凌晨壓縮機跳掉、夜班只有保全
> I want to 5min 內遠端確認、即時告警有人能處理
> So I can 不再賠 80 萬、不再被董事會詢問

**JTBD 2**：
> When 客戶端（全家）要求溫度履歷
> I want to 客戶端能自己 QR 掃出當批溫度
> So I can 不再被客戶扣款、保住 1500 萬年訂單

**JTBD 3**：
> When 2027 上市櫃送件
> I want to 有 6 個月可信稽核資料
> So I can IPO 順利、達到家族傳承目標

→ 三個 JTBD 都是 push 力強的場景、提案就會打中。

## 跟其他方法的關係

| 方法 | 跟 JTBD |
|---|---|
| Kano Model | JTBD 找 job、Kano 分類功能、互補 |
| Persona | Persona 描述「誰」、JTBD 描述「為什麼」、JTBD 更深 |
| User Story | 「As X, I want Y, so that Z」≈ JTBD 句型、JTBD 更強調情境 |
| Design Thinking | JTBD 是 Empathize 階段的工具 |

## 常見錯誤

- ❌ 把客戶嘴上講的功能當 Job
- ❌ 只挖 Functional、漏 Emotional / Social
- ❌ 用 JTBD 句型但訪談太短、深度不夠
- ❌ 沒用 4 力分析、不知道為什麼客戶簽不下去
