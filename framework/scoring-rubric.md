# 提案品質 Scoring Rubric

`scripts/completeness-check.sh` 只是 keyword match。
這份是**真正的品質評估標準**、給人工 review 用。

## 100 分總分配置

| 大項 | 滿分 | 重點 |
|---|---|---|
| 1. 痛點挖掘深度 | 15 | 是否挖到真痛點、不是表面需求 |
| 2. 競品 / 替代方案處理 | 10 | 有沒有點出客戶選擇空間 |
| 3. 方案結構與差異化 | 15 | 是否 ≥ 3 方案、有 Win Themes |
| 4. 架構與技術深度 | 10 | 圖清楚、NFR 量化、整合明確 |
| 5. 合規 / 資安 | 10 | 對應法規 / 標準完整度 |
| 6. 商務 / 報價 | 10 | 拆項、付款、TCO、不含項 |
| 7. 驗收 / 效益 | 10 | 量測標準、基線、KPI 對應 |
| 8. 風險 / 反方 | 10 | 風險清單、責任邊界、反方審查 |
| 9. 強制檢查項覆蓋 | 10 | A-H 8 項 |

每項依**Excellent / Good / Marginal / Poor** 4 級給分。

---

## 1. 痛點挖掘深度（15 分）

| 級別 | 分數 | 標準 |
|---|---|---|
| Excellent | 13-15 | 痛點明確量化（頻率 × 影響 = 年化）、覆蓋產銷人發財 ≥ 3 個 function、用了 JTBD 3 層、有客戶引述 |
| Good | 10-12 | 痛點量化、覆蓋產銷人發財 ≥ 2 個 function、含客戶實例 |
| Marginal | 6-9 | 痛點有量化但只覆蓋 1 個 function、靠推斷不靠訪談 |
| Poor | 0-5 | 痛點空泛（「效率不佳」「需要改善」）、無量化、無訪談證據 |

## 2. 競品 / 替代方案處理（10 分）

| 級別 | 分數 | 標準 |
|---|---|---|
| Excellent | 9-10 | ≥ 3 個競品 + 替代方案、SWOT 每個競品、定位四象限圖、客戶選擇樹 |
| Good | 7-8 | ≥ 2 個競品、含「維持現狀」替代 |
| Marginal | 4-6 | 1 個競品比較 / 只列功能對照 |
| Poor | 0-3 | 沒競品分析 / 把對方講爛 / 競品放提案末尾 |

## 3. 方案結構與差異化（15 分）

| 級別 | 分數 | 標準 |
|---|---|---|
| Excellent | 13-15 | 3 方案 + 明確推薦、Win Themes 3-5 個、Discriminators 可驗證、分階段路線 |
| Good | 10-12 | 2-3 方案、有推薦、有差異化論述 |
| Marginal | 6-9 | 只給 1 方案 / 三方案差異不明顯 |
| Poor | 0-5 | 沒方案結構 / Win Themes 空泛 / 沒分階段 |

## 4. 架構與技術深度（10 分）

| 級別 | 分數 | 標準 |
|---|---|---|
| Excellent | 9-10 | 4 張圖（業務 / 資料 / 應用 / 技術）+ C4 補強、NFR 8+ 條量化、整合介面表雙方確認 |
| Good | 7-8 | 2-3 張圖、NFR 量化、整合表 |
| Marginal | 4-6 | 1 張圖塞 4 層 / NFR 寫但不量化 |
| Poor | 0-3 | 沒架構圖 / NFR 全部「依需求」「視情況」 |

## 5. 合規 / 資安（10 分）

| 級別 | 分數 | 標準 |
|---|---|---|
| Excellent | 9-10 | NIST CSF 6 功能對齊、適用法規矩陣、SBOM 附 JSON、Pen Test 報告承諾、DPA 附錄 |
| Good | 7-8 | 主要法規對應、資安政策、SBOM 承諾提供 |
| Marginal | 4-6 | 「我們符合 ISO 27001」沒證書、「加密」沒寫演算法 |
| Poor | 0-3 | 沒合規章節 / 法務一問破功 |

## 6. 商務 / 報價（10 分）

| 級別 | 分數 | 標準 |
|---|---|---|
| Excellent | 9-10 | 拆 ≥ 5 大項、付款 3 段以上、TCO 3 年、用量上限明確、不含項清楚 |
| Good | 7-8 | 拆 3-4 項、付款條件清楚、有 TCO |
| Marginal | 4-6 | 一個總價、付款 2 段、無 TCO |
| Poor | 0-3 | 「依實際工時計算」 / 沒寫「不含」 / 沒匯率稅金條款 |

## 7. 驗收 / 效益（10 分）

| 級別 | 分數 | 標準 |
|---|---|---|
| Excellent | 9-10 | 驗收項可測 / 可重現 / 有時限、基線已量、KPI 對應業務目標、30/60/90 回顧計畫 |
| Good | 7-8 | 驗收量化、KPI 對應、有基線 |
| Marginal | 4-6 | 驗收標準有但模糊（「使用順暢」）/ 沒基線 |
| Poor | 0-3 | 「客戶滿意」 / 沒可測標準 |

## 8. 風險 / 反方（10 分）

| 級別 | 分數 | 標準 |
|---|---|---|
| Excellent | 9-10 | ≥ 10 條風險、前 5 條有對策、責任邊界 RACI、反方審查紀錄附在末尾 |
| Good | 7-8 | ≥ 5 條風險、前 3 條有對策、責任邊界 |
| Marginal | 4-6 | 3-5 條風險、無對策 / 籠統 |
| Poor | 0-3 | 「無風險」 / 「皆已掌握」 / 沒反方審查 |

## 9. 強制檢查項覆蓋（10 分）

每個 mandatory check 1.25 分（A-H 共 8 項）：

| Check | 標準 |
|---|---|
| A RTM | 雙向追溯、每痛點 → 功能 → 驗收 → 報價可追 |
| B 溝通計畫 | 頻率 / 方式 / 紀錄 / Escalation Path |
| C 決策紀錄 | ADR 機制、CR 流程 |
| D Secure SDLC | 7 階段每段有寫 |
| E SBOM | 格式（SPDX / CycloneDX）、License、Sub-processor |
| F RTO / RPO | 明確數字、依據、演練 |
| G 退出機制 | 資料匯出、刪除、移轉協助 |
| H 證據包 | ≥ 3 case、reference 名單、認證 |

---

## 評分執行方式

### 自評（提案送出前）

```
1 個 reviewer、1 小時跑完
總分 < 70：回去補
總分 70-85：可送出但內部審查
總分 85+：可直接送
```

### 同儕互評（推薦）

```
2 個 reviewer 獨立評
差距 > 10 分：第三人仲裁
平均 < 75：回去補
```

### 客戶模擬評分

模擬 3 種客戶（老闆 / IT / 法務）各跑一次：

| 客戶角色 | 重點看 |
|---|---|
| 老闆 | 1、3、6、7 大項 |
| IT | 4、5、9 大項 |
| 法務 | 5、6、8、9 大項 |

→ 三種視角總分平均、低於 75 必補。

---

## 加分 / 減分項

### 加分（最多 +10）

- 客戶引述（同產業）+2
- Discriminator 可驗證（有 source）+2
- 反方審查由外部視角執行 +2
- 同產業 case ≥ 5 個 +2
- 客戶 IT 已 pre-review 過 +2

### 減分（無上限）

- 任何具體主張沒驗 source -5
- 「絕對」「100%」「完美」-3
- 沒有 RTO/RPO -5
- 沒有 SBOM -3
- 沒有退出機制 -5
- 競品在末尾才出現 -3
- 沒寫「不含」項 -3

---

## 跟 `scripts/completeness-check.sh` 的差別

| | completeness-check.sh | scoring-rubric.md |
|---|---|---|
| 方式 | Keyword match | 人工評分 |
| 深度 | 表面（有出現 keyword 就 ✓）| 評**質量**（深淺、可驗證、契合度）|
| 速度 | < 1 秒 | 1-3 小時 |
| 用法 | 自動化、過 / 不過 | 真實品質判斷 |

→ 兩個都跑：先過 completeness-check、再用 rubric 細評。
