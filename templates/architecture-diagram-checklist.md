# 架構圖 Checklist

每份提案至少要有 4 張圖，不要混在一張畫。

## 一張圖的必有元素

不論哪一層，每張圖都要含：

- [ ] 標題（這是哪一層）
- [ ] 版本號 / 日期
- [ ] 圖例（符號、顏色、線型）
- [ ] System Boundary（系統範圍）
- [ ] External Actors（外部使用者、第三方）
- [ ] 流向（箭頭）
- [ ] 簡短說明（每個元件 1-2 句）

## 四層圖

### 1. 業務架構圖（Business Architecture）

回答：「業務上發生什麼事？」

必含：

- [ ] 業務能力 / 流程區塊
- [ ] 涉及角色（部門 / 人）
- [ ] 觸發事件（什麼時候啟動）
- [ ] 輸入 / 輸出
- [ ] 跨組織邊界（公司、部門）

**範例元素**：訂單 → 生管 → 製造 → 品管 → 出貨

### 2. 資料架構圖（Data Architecture）

回答：「資料從哪來、到哪去、長什麼樣？」

必含：

- [ ] 主要資料實體（訂單、客戶、產品、設備）
- [ ] 資料來源（外部系統、感測器、人輸入）
- [ ] 資料儲存（DB、Data Lake、快取、佇列）
- [ ] 資料流（哪個系統 → 哪個系統）
- [ ] 主資料 vs 交易資料
- [ ] 資料分類（敏感度標記）
- [ ] 保留期 / 生命週期

### 3. 應用架構圖（Application Architecture）

回答：「系統元件怎麼組？」

必含：

- [ ] 各應用 / 服務 / 元件
- [ ] 內部 API / 介面
- [ ] 外部 API / 介面（與第三方）
- [ ] UI 層（Web、App、API）
- [ ] 後端服務
- [ ] 整合中介（ESB、API Gateway、訊息佇列）
- [ ] 與既有系統的對接點

### 4. 技術架構圖（Technology Architecture）

回答：「東西實際跑在哪？」

必含：

- [ ] 雲端 / 地端 / 混合
- [ ] Region / AZ / VPC
- [ ] 網路（subnet、SG、WAF、CDN）
- [ ] 運算（EC2、ECS、Lambda、K8s）
- [ ] 儲存（RDS、S3、DynamoDB、Redis）
- [ ] 安全層（IAM、KMS、Secrets Manager）
- [ ] 監控 / 日誌
- [ ] 部署管道（CI/CD）

## 補充常見圖（依需要）

| 圖名 | 何時加 |
|---|---|
| 部署圖（Deployment） | 跨地、跨雲、混合 |
| 序列圖（Sequence） | 複雜整合 |
| 狀態圖（State） | 訂單、案件狀態流轉 |
| 拓樸圖（Network Topology） | 工廠、醫院、複雜網路 |
| 威脅模型圖（STRIDE） | 高資安要求案件 |
| 服務藍圖 | 模組 04 客戶服務流程 |

## 常見地雷

- ❌ 一張圖塞 4 層 → 看不懂
- ❌ 沒標 System Boundary → 範圍模糊
- ❌ 沒標版本 / 日期 → 改版混亂
- ❌ 用截圖嵌入 PPT → 模糊、改不了
- ❌ 圖例顏色憑感覺 → 客戶看不出哪是我們、哪是既有
- ❌ 元件名稱不一致 → A 圖叫「訂單服務」、B 圖叫「OMS」

## 驗收

寫完後檢查：

- [ ] 客戶 IT 看得懂嗎（不只是我們看得懂）
- [ ] 客戶老闆看得懂嗎（業務圖至少要懂）
- [ ] 跟 RTM 對得起來嗎
- [ ] 跟 NFR 對得起來嗎
- [ ] 跨團隊用同一張嗎（PPT、提案、合約附錄）
