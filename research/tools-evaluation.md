# 工具評估方法論

承襲 V8 工具研究版。
給未來「要不要採用某個工具 / Skill / 套件」的評估流程。

## 為什麼需要這個

實務上，每次想加工具都會碰到：

- 看到 GitHub star 多就想用
- 看到 demo 好看就想用
- 看到別人公司在用就想用

這些都不夠。要有一套系統評估方法。

## 工具評估五階段

```
1. 候選收集 → 2. 初篩 → 3. 評分表 → 4. PoC → 5. 採用決策
```

### 階段 1：候選收集

來源：

- GitHub search（star、近期更新）
- Awesome lists
- Hacker News、Product Hunt、Reddit
- 行業聚會、研討會
- 客戶推薦
- 競品在用

不要只看一個來源 → 各自有偏見。

### 階段 2：初篩 Checklist

每個候選過 5 個 hard filter：

- [ ] License 可商用（MIT / Apache 2.0 / 商業）
- [ ] 過去 6 個月有更新
- [ ] 有正式文件（不是只有 README）
- [ ] Issue 處理頻率合理（不是死庫）
- [ ] 有公開使用案例（不只我們用）

任一 fail → 淘汰。

### 階段 3：評分表（Scorecard）

通過初篩的候選做評分表，每項 1-5 分：

| 維度 | 子項 | 權重 |
|---|---|---|
| **功能契合度** | 必需功能覆蓋、選用功能覆蓋、擴展性 | 30% |
| **品質** | 程式碼品質、文件品質、測試覆蓋 | 20% |
| **社群與支援** | star、貢獻者、issue 處理、商業支援 | 15% |
| **License 與商業** | License 風險、商業條款、供應商風險 | 10% |
| **資安** | 漏洞紀錄、SBOM、安全 advisory | 10% |
| **整合難度** | 與既有系統契合、學習曲線 | 10% |
| **總擁有成本** | 授權、運維、培訓、遷出成本 | 5% |

加權後 **≥ 3.5 進階段 4，< 3.5 淘汰**。

### 階段 4：PoC

候選的 2-3 個跑 PoC：

- 真實場景（不是 demo）
- 1-2 週時限
- 量化結果（時間、錯誤率、客戶滿意度）
- 列出實際遇到的雷

### 階段 5：採用決策

| 因素 | 內容 |
|---|---|
| PoC 結果 | 量化指標 |
| 評分表分數 | 階段 3 結果 |
| 戰略契合度 | 公司技術方向 |
| 替代風險 | 不採用會怎樣 |
| 退出成本 | 採用後再換的成本 |

決策不只看分數，也看戰略。

## 工具類別與評估重點

### Claude Skill / Plugin

| 評估點 | 內容 |
|---|---|
| Skill 安全 | 用 `/security-scan` 掃 |
| 觸發詞合理 | 不會被誤觸 |
| 文件清晰 | README、example |
| 維護者可信 | GitHub 帳號歷史 |
| 更新頻率 | 近期活躍 |

### Open Source Library

| 評估點 | 內容 |
|---|---|
| License | 對應 [SBOM 章節](../framework/8-mandatory-checks/E-sbom-license-supply-chain.md) |
| 維護者 | 個人 vs 組織 vs 基金會 |
| 競品 | 是否有更好的替代 |
| 鎖定風險 | 換掉容不容易 |

### SaaS 服務

| 評估點 | 內容 |
|---|---|
| 訂閱條款 | 一年合約 vs 月退 |
| 資料主權 | 區域、退出 |
| SOC 2 / ISO | 認證 |
| 串接 API | 完整 / 部分 |
| 客戶服務 | 中文、時區 |

### AI 模型 / 服務

| 評估點 | 內容 |
|---|---|
| 模型版本管理 | 是否會自動切換 |
| 資料訓練 | 客戶資料是否拿去訓練 |
| 評估指標 | 準確率、延遲、成本 |
| Fallback | 備援模型 |
| Prompt 版控 | 是否支援 |

## 評估產出

每次評估產出：

```
# [工具名稱] 評估報告

## 評估對象
- 名稱：
- 版本：
- License：
- URL：

## 候選收集（其他 N 個競品）

## 初篩結果
- [ ] License
- [ ] 維護
- [ ] 文件
- [ ] Issue
- [ ] 案例

## 評分表（總分 / 5）

## PoC 結果
- 場景：
- 結果：
- 雷點：

## 採用決策
- 採用 / 不採用 / 觀察
- 理由：

## 評估日期 / 評估者
```

報告存放 `research/evaluations/`，未來查得到。
