# 個人能力建構

ProposalOS 是流程，但流程靠人跑。
這份是「使用者自我能力訓練」playbook，**不是提案內容的一部分**。

## 顧問式提案需要的核心能力

| 能力 | 為什麼重要 | 怎麼練 |
|---|---|---|
| 產業敏感度 | 看出客戶產業的隱藏邏輯 | 每週讀 3-5 份產業報告 |
| 訪談技巧 | 把客戶嘴上的解法問成真痛點 | 5 次「為什麼」、結構化訪談 |
| 結構化思考 | 把混亂資訊變 13 模組 | 每案做完寫一份結構化覆盤 |
| 寫作能力 | 客戶讀得懂、老闆愛看 | 多寫、給人改、自己改 |
| 簡報能力 | 30 秒抓住老闆、2 小時 close 案 | 多講、錄影自我檢討 |
| 商務直覺 | 知道怎麼報、怎麼拆 | 每案分析報價結構與成交細節 |
| 法務常識 | 看得懂客戶法務的擔心 | 讀過 5 份 DPA、SLA、SOW |
| 資安常識 | NIST CSF、SDLC、SBOM 不是名詞 | 跟我方 CISO 對話、實際參與 |
| AI 使用力 | AI 是工具不是魔法 | 每天用、找出 AI 出錯的時候 |

## 每案復盤模板

每個提案結束（不論成交與否）寫一份覆盤：

```
## 案件名稱

## 結果
- [ ] 成交 / 未成交 / 暫緩

## 提案歷程

### 接觸到簽約（時間軸）
- D+1：
- D+7：
- ...

### 關鍵決策
- 決策 1：選了 A 而非 B，因為 ...
  - 結果：對 / 錯，學到 ...

### 客戶反應
- 我說了 X，客戶 reaction Y
- 我以為 A，實際是 B

## 對照 13 模組
- 哪幾個模組做得好？
- 哪幾個模組做得不夠？

## 對照 8 強制檢查項
- 哪些有做？
- 哪些被客戶問住？

## 反方審查
- 客戶實際問了什麼？
- 哪些是我們預想到、哪些是沒預想到？

## 下次怎麼改

### 訪談階段
### 提案階段
### 商務階段
### 維運階段

## 加進證據包
- 案例摘要：
- 客戶引述：（如有授權）
- 可作 reference：是 / 否
```

## 錯誤案例庫

把「曾經出錯」的案例累積成庫，每年讀一次。

| 案例 | 出錯點 | 學到 |
|---|---|---|
| | | |

範例：

| 案例 | 出錯點 | 學到 |
|---|---|---|
| XX 食品工廠 | RTO 寫 4hr 但實際 8hr | RTO 要實測再寫，不要拿行業數字 |
| XX 銀行 | 沒先確認雲端委外要報准 | 金融業先讀委外辦法 |
| XX 醫院 | 沒識別 SaMD 風險 | 醫療臨床決策功能必先諮詢法規 |

## 提案審查 Checklist（給自己）

提案送出前自我審查：

### 13 模組
- [ ] 01 產業：有寫過去 3-5 年趨勢，引用源可驗
- [ ] 02 競品：3 個選項以上，含「維持現狀」
- [ ] 03 問題：每個痛點有量化、對應 KPI
- [ ] 04 流程：As-Is / To-Be 都有，含例外流程
- [ ] 05 角色：RACI 表覆蓋 ≥ 5 個活動
- [ ] 06 資料：來源 / 治理 / 整合都寫
- [ ] 07 合規：法規、資安、AI（若有）都有
- [ ] 08 架構：4 張圖（業務 / 資料 / 應用 / 技術）
- [ ] 09 方案：3 個選項，有推薦
- [ ] 10 報價：拆項、付款、TCO
- [ ] 11 驗收：可量測、含基線
- [ ] 12 風險：≥ 10 條風險、前 3 條有對策

### 8 強制檢查項
- [ ] A RTM：雙向可追
- [ ] B 溝通計畫：含 escalation
- [ ] C 決策紀錄：機制定義
- [ ] D Secure SDLC：每階段都有
- [ ] E SBOM：附 JSON 或承諾提供
- [ ] F RTO/RPO：明確數字、有依據
- [ ] G 退出機制：資料、刪除、移轉
- [ ] H 證據包：≥ 3 個 case、reference

### 提案品質
- [ ] 沒有沒驗的具體主張（數字、廠商、案例）
- [ ] 沒有「絕對」、「100%」、「完美」
- [ ] 有量化效益
- [ ] 老闆讀 1 頁能懂全案
- [ ] 工程師讀架構能懂技術
- [ ] 法務讀條款能簽

## 學習資源

### 必讀書

| 書 | 內容 |
|---|---|
| 《SPIN Selling》| 顧問式銷售 |
| 《Challenger Sale》| 教育型銷售 |
| 《Crossing the Chasm》| 技術產品市場 |
| 《Continuous Discovery Habits》| 持續訪談 |
| 《Software Architecture Patterns》| 架構 |
| 《Designing Data-Intensive Applications》| 資料密集系統 |
| 《Phoenix Project》| DevOps |
| 《Accelerate》| DevOps 數據 |

### 必懂框架

- BANT / MEDDIC / MEDDPICC（銷售）
- Jobs-to-be-Done（需求）
- IIBA BABOK（業務分析）
- TOGAF（企業架構）
- NIST CSF / RMF（資安）
- ITIL 4（服務管理）
- PMBOK / PRINCE2（專案管理）
- ADKAR / Kotter（變革）

### 跟對的人

- 公司內部：找做過 5+ 案的人請教
- 公司外部：產業協會、行業會議
- 同行：定期交流（不洩客戶資訊）

## 自我節奏

- 每天：早上 30 分讀產業新聞
- 每週：寫一份簡短覆盤（任何工作學到的事）
- 每月：跑一次自我提案審查 checklist
- 每季：讀完一本必讀書 / 一份大型行業報告
- 每年：完整覆盤過去 12 個月案件，更新錯誤案例庫
