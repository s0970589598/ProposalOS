# AI 模擬訪談 — 3 模式分層使用

⚠️ **本檔處理一個敏感議題**：什麼時候可以用 AI 模擬客戶 / 評委 / 業務員 / 其他訪談對象的回答、什麼時候**絕對不可以**。

## 為什麼需要這份指引

[ProposalOS GCP retrospect](../examples/goodlinker-gcp-pitch-retrospective.md) AP-NEW-01 指出「客戶引言段 placeholder 未訪談」是真實雷。
但訪談需要約時間、有時 deadline 緊、是否能用 AI 模擬替代？

→ 答案：**部分可以、部分不可以**。本檔規範三種模式各自的紅線。

---

## 3 種模式

### 模式 A：真實訪談（最高品質）

| 項目 | 內容 |
|---|---|
| 對象 | 真人 |
| AI 角色 | 整理紀錄、不創造內容 |
| 進提案 | ✅ 可、含客戶授權引言 |
| 流程 | [interview-script.md](interview-script.md) |
| 信度 | 高 |
| 成本 | 約 1.5-2 hr / 場 + 約時間 |

### 模式 B：AI 預判 + 真實訪談（推薦標準）

| 項目 | 內容 |
|---|---|
| 對象 | 真人 |
| AI 角色 | 訪前**預測**對方可能說什麼 + 訪後比對「預判 vs 實際」 |
| 進提案 | ✅ 真實答案進、預判紀錄留作學習 |
| 流程 | 本檔下方詳述 |
| 信度 | 高（含學習迴路）|
| 成本 | A 模式 + 30 min AI 預判 |

**為什麼推薦這個**：用 AI 增強真實訪談、不是取代。

### 模式 C：純 AI 模擬訪談（內部用、有紅線）

分兩種子模式：

#### C1：純內部準備 / Brainstorm（安全）

| 項目 | 內容 |
|---|---|
| 對象 | 無真人 |
| AI 角色 | 扮演 persona 推論 |
| 進提案 | ❌ **絕對不可** |
| 用途 | 內部找盲點、設計問題、訓練訪談者 |
| 紅線 | AI 模擬內容不能離開內部準備檔 |

✅ 完全可以做。

#### C2：當提案內容 placeholder（高風險）

| 項目 | 內容 |
|---|---|
| 對象 | 無真人 |
| AI 角色 | 推論回答、填提案空格 |
| 進提案 | ⚠️ **強烈不建議**、必須含警示標頭 + 替換計畫 |
| 用途 | 沒對象、deadline 緊、暫時 placeholder |
| 紅線 | 多個（見下方）|

❌ 預設**不要**做。例外見下方。

---

## C2 的 4 個紅線

如果 deadline 緊、一定要 C2、**4 個紅線都不可越**：

### 紅線 1：必含警示標頭

```markdown
### 標竿客戶 1: 海瑞食品

> ⚠️ **以下為 AI 推論版引言**（基於公開資料 + 食品業典型 pattern）、
> **待真實訪談驗證**。Pitch 簽核前必訪 1 場、用真實引言取代。
>
> 「[AI 推論的話]」
> — _推論依據：sustained 8.4 年 0 改、食品業典型 onboarding pattern_
> — _推論日期：YYYY-MM-DD_、_預計訪談日：YYYY-MM-DD_
```

→ 客戶 / 評委看到知道這不是真實 quote。

### 紅線 2：必有「替換計畫」

每段 C2 內容必有 deadline + 負責人 + 替換動作：

```markdown
**替換計畫**：6/8 訪海瑞、6/9 取代為真實引言、6/10 送 pitch
**負責人**：Allen
**追蹤狀態**：⏳ 待訪
```

→ 不能變成「永遠 placeholder」。

### 紅線 3：絕不用具體 quote 風格

❌ **不准**這樣：
> 「我們當初是憑經驗設規則、8.4 年不需要改的真因是 ___」 — 海瑞廠長（AI 推論）

✅ **改成**：
> AI 推論該訪談可能挖到以下主題：
> - 規則設計過程（典型回答：憑經驗 + 慢慢加）
> - 8.4 年穩定真因（典型：產線穩定 + 老師傅）
>
> 真實 quote 待 6/8 訪談後填入。

→ 看起來像 quote 的東西讀者會當 fact、不要寫像 quote 的句子。

### 紅線 4：絕不放在「客戶引言段」標題下

`### 客戶引言：`段下放 C2 內容、讀者會直接信。
改放在 `### 訪談規劃` / `### Pending Customer Validation` 等明顯非引言的段。

## 模式 B 標準流程

### Step 1：訪前 AI 預判（30 min）

用 [AI 訪談模擬 template](../templates/ai-interview-simulation-template.md) + `proposal-os` skill 觸發：

```
觸發詞：「幫我預判 [客戶名] 會怎麼回 [主題]」

AI 動作：
1. 蒐集客戶公開資料 + 已知 internal data（pitch / RTM / 客戶 baseline）
2. 對每個主問題、給 3 個信度等級的預判：
   - H（高信度）：典型 pattern、很可能對
   - M（中信度）：常見變體
   - L（低信度）：可能性、但很多客戶不會這樣回
3. 標出「反向預判」（客戶可能不會 / 不能回答的點）
4. 標出「高價值題」（最可能挖到真實金句、訪談者要留時間）
```

### Step 2：訪前準備（追問策略）

依預判結果設計：

- H 預判題 → 表面確認即可、不花太多時間
- M / L 預判題 → 留追問深度
- 反向預判 → 從側面問、不要逼客戶迴避
- 高價值題 → 訪談時間配置優先

### Step 3：真實訪談（依 [interview-script.md](interview-script.md)）

正常跑、預判**不影響**訪談本身。

### Step 4：訪後比對（24 hr 內）

填預判 template 的「實際」+ 「差異」欄：

| 差異類型 | 處理 |
|---|---|
| AI 預判 H 但實際完全不同 | **AI 盲點**、是 anti-pattern 候選 |
| AI 預判到、實際也對 | 證明 AI 在這類有信度 |
| AI 沒預判到、實際很重要 | 訪談獨特價值的證據 |

### Step 5：累積學習

把 Step 4 的 anti-pattern 候選加進 [anti-patterns.md](../framework/anti-patterns.md)。

範例（從 GCP pitch 海瑞訪談得到）：

```
AP-NEW-XX：AI 預判廠長會說「sustained 4 階是自己設的」、
實際是 GoodLinker AM 設、客戶不記得 sustained 機制
→ AI 容易高估客戶對技術細節的記憶 / 知識
→ 訪談這類細節題要透過 AM 或紀錄、不要直接問客戶
```

## 何時用哪個模式

```
有真實對象 + 時間充足   → 模式 A 或 B（推薦 B）
有真實對象 + 時間緊     → 模式 B（AI 預判替訪前準備省時間）
無真實對象 + 內部用     → 模式 C1（安全）
無真實對象 + 要 ship    → ⚠️ 慎用 C2、4 紅線必守
```

## C1 範例用途（安全的 AI 模擬）

- 訪談前自己 brainstorm「客戶可能會問什麼難題」
- 內部反方審查模擬「假設客戶法務看到合約會怎麼想」
- 業務新人訓練「假裝你是難搞客戶」
- ProposalOS skill 觸發前先 dry run

→ 這些都不會進客戶提案、純內部用、可放心。

## C2 範例（真實壞用法、不要做）

- 「沒約到海瑞、用 AI 寫一段海瑞會說的話、放進 pitch 客戶引言段」
- 「業務都沒空訪、用 AI 推論 deal-loss 對手、寫進競品分析」
- 「PEST 分析的『業界都這樣做』我們用 AI 推論、沒查 source」

→ 違反 [CLAUDE.md](../CLAUDE.md) 「外部事實主張驗證」紀律。

## 跟其他 Skill 的關係

| Skill | 角色 |
|---|---|
| `proposal-os` | 主流程、提案編寫 |
| `cove-verify` | 驗證具體 claim、特別對 C2 內容 |
| `pair-review` | Fresh-context 視角、可發現 AI 模擬味 |
| `market-research` | 訪談前蒐集公開資料、增強 AI 預判信度 |

模式 B 推薦 pipeline：

```
market-research（蒐集公開資料）
   ↓
proposal-os「幫我預判 X」（B 模式預判）
   ↓
真實訪談
   ↓
cove-verify（核對訪談裡的具體 claim）
   ↓
寫進提案
```

## 對 ProposalOS 框架的長期意義

- **訪談累積**（A 模式）：累積真實客戶聲音、anti-patterns
- **預判累積**（B 模式）：累積「AI 預判 vs 實際」差異、AI 盲點清單
- **C1 累積**：累積內部 brainstorm 模板、訪談者訓練材料
- **C2 紀錄**：每次 C2 使用必紀錄、追蹤替換進度、防止 placeholder 永久化

→ B 模式累積 6-12 個月、會出現「AI 對食品業 X 類問題預判信度高、對 Y 類差」的 pattern、進化成更精準的訪談準備工具。
