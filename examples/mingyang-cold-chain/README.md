# 明陽食品冷鏈監控提案（虛構範例）

⚠️ **虛構案例 — 用於教學示範，非真實客戶**

## 案例背景

| 項目 | 內容 |
|---|---|
| 客戶（虛構） | 明陽食品股份有限公司 |
| 產業 | 食品 / 冷鏈加工 |
| 規模 | 員工 200 人、年營收 5 億 |
| 場景 | 3 個冷凍倉、12 台冷藏配送車、月配送量 500 噸 |
| 客戶端 | B2B 餐飲通路（早午餐連鎖、便利商店、超市） |
| 提案類型 | 新建 IoT 冷鏈監控 + 食品履歷系統 |
| 預算範圍 | 300-500 萬 |
| 時程 | 6 個月內上線 |
| 我方接觸者 | 王副總（營運副總，案件 Champion） |

## 利害關係人

| 角色 | 關心 | 立場 |
|---|---|---|
| 張董（董事長）| ROI、品牌風險、上市櫃準備 | Economic Buyer |
| 王副總（營運）| 食安事故、客訴、營運效率 | Champion |
| 李經理（IT）| 整合、運維負擔、資安 | Technical Buyer / 可能 Blocker |
| 廠長（生產）| 不要影響產線 | User Buyer |
| 物流主管 | 司機抗拒新工具 | User Buyer |
| 法務 | 食安法、客戶合約 | I (Informed) |

## 客戶現況痛點

| 痛點 | 發生頻率 | 單次影響 | 年化影響 |
|---|---|---|---|
| 冷凍倉壓縮機故障，2-3 小時後才發現 | 每月 1-2 次 | 損失 20-50 萬（產品報廢 + 客訴）| 400-1200 萬 |
| 配送車到貨時客戶質疑溫度，沒記錄 | 每週 1-2 次 | 客戶扣款 5-10 萬 | 250-500 萬 |
| 食品履歷靠 Excel 手填，客戶稽核 1 天才能查到批號 | 每月 5-8 次 | 客戶抱怨 / 失去訂單 | 難量化 |
| 客訴出問題回追原料批次要 1-2 天 | 每季 1-2 次 | 召回成本 100+ 萬 | 200-800 萬 |

## 經營目標對應

| KPI | 現況 | 目標（12 個月內） |
|---|---|---|
| 冷鏈斷鏈次數 | 12-24 次/年 | < 3 次/年 |
| 配送溫度爭議客訴 | 50-100 次/年 | < 10 次/年 |
| 食品履歷查詢時間 | 1 天 | < 5 分鐘 |
| 客戶稽核通過率 | 80% | > 95% |
| 上市櫃食安系統合規 | 未達標 | 達標 |

## 已評估選項（客戶端）

| 選項 | 評估結果 | 為什麼沒選 |
|---|---|---|
| ERP 廠商 A 內建模組 | 看過 demo | 沒有 IoT 採集、只能輸入 |
| 國際大廠 B 完整方案 | 看過報價 | 1500 萬太貴、3 年導入太久 |
| 自己開發 | 內部討論過 | IT 只有 2 個人、做不來 |
| 維持現狀 | — | 上市櫃前必須改善 |

## 競品分析

| 競品 | 切入角度 | 客戶感受 |
|---|---|---|
| GoodLinker（我方）| IoT + 平台 + 中型客戶經驗 | 推薦 |
| 競品 A | 純 IoT 硬體商 | 沒有後台 / 報表 |
| 競品 B | 國際 SaaS | 太貴、不貼台灣食安法 |

## 案件結論（示範）

- 推薦方案：**標準版 NT$ 380 萬**（含 1 年支援）

- 預估回本：14 個月
- 主要風險：李經理（IT）對運維增量的擔心 → 用「我方接手 1 年運維」化解
